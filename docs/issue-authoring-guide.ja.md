# Issue作成・課題調査ガイド

リファクタリング計画(#60〜#77)とプロダクト価値計画(#79〜#94)の作成プロセスから抽出した、**「ジュニアが実装してもシニアと同じ結果になる issue」を量産するための手順とチェックシート**です。Mockport 固有の内容は例として登場するだけなので、他リポジトリにもそのまま流用できます。

---

## 0. 大原則(これだけは崩さない)

1. **証拠なき指摘を書かない** — すべての指摘は `file:line` と再現コマンド(grep / diff / 実行ログ)に裏付けられる。「〜のはず」「一般的に〜」で書いた瞬間に、実装者が現地で迷子になる
2. **網羅は根性ではなく構造で作る** — 「怪しい所を探す」のではなく、後述の探索レンズ(主張vs実装 / 同種の並置 / 能力vs使用)を機械的に当てる。レンズが正しければ、誰がやっても同じ発見に到達する
3. **判断は issue に埋め込み、実装者に委ねない** — 設計の分岐(AかBか)はシニアが理由付きで決め切って書く。委ねてよいのは「手順どおりやって出てくる差分」だけ
4. **正直さを最優先** — silent failure・宣言済み未実装・ドリフトした文書は、機能不足よりも先に潰す(信頼はプロダクトの土台)
5. **検証は発明しない** — 受け入れ基準の検証コマンドは CI と同一のものを使う。CI に無い検証が必要になったら、それ自体が issue 候補

---

## 1. 全体フロー

```
Stage 0          Stage 1            Stage 2         Stage 3          Stage 4         Stage 5
コンテキスト把握 → 構造的調査       → 証拠固め      → 計画設計       → 執筆          → 投稿・配線
(30分-1時間)     (レンズを当てる)   (spot-check)    (フェーズ分け)   (テンプレート)  (相互参照)
```

時間配分の目安: 調査・検証に全体の **6割** を使う。執筆が先行して調査が後追いになったら手順違反(書きながら裏取りすると、裏が取れなかった指摘を捨てられなくなる)。

---

## 2. Stage 0: コンテキスト把握

コードを精読する前に、**プロジェクトが自分について何を語っているか**を30分で収集する。

- [ ] README / ROADMAP / CHANGELOG を読む(=プロダクトの「主張」リスト。後で実装と突合する)
- [ ] 言語・ランタイムのバージョンを確認(`go.mod` / `package.json` 等)→ 「使えるのに使っていない機能」の基準線になる
- [ ] CI 定義(`.github/workflows/`)と Makefile を読む → 検証コマンドの正をここから取る
- [ ] 既存の issue / PR / ラベル / マイルストーンを確認(`gh issue list --state all`、`gh label list`)→ **既存の計画体系があれば、それに接続する**(新しい体系を発明しない)
  - 実例: リポジトリに `phase-1`〜`phase-5` ラベルが既に定義されていたため、リファクタリング issue はその構成に従った
- [ ] ポリシー文書(CONTRIBUTING / docs配下の policy 系)を読む → 「意図的な未実装」と「単なる欠落」を区別する材料
  - 実例: `scenarios:` 設定の無視は scenario-policy.md で「future work」と明言済みだったため、issue は「本実装」ではなく「警告で正直になる」に設計した
- [ ] ファイル規模の地図を作る: `find . -name "*.go" | xargs wc -l | sort -rn | head -25`(突出したファイルは必ず何かある)

## 3. Stage 1: 構造的調査(探索レンズ)

### レンズA: 「主張」vs「実装」の突合(嘘・ドリフトの発見)

Stage 0 で集めた主張を1つずつコードで検証する。**最も価値の高い発見はほぼここから出る。**

| 主張の在処 | 検証方法 | 実例(今回の発見) |
|---|---|---|
| メタデータ・設定スキーマの宣言 | 宣言を消費するコードを grep。呼び出しゼロ=嘘 | `Reset: true` 宣言に対し `grep "\.Reset("` → 本番呼び出し0件(#79) |
| 設定ファイルで受理されるキー | パース後の参照箇所を grep。ゼロ=silent ignore | `cfg.Scenarios` の消費箇所0件(#81) |
| シナリオ・モード名の意味 | 名前が示す挙動を実装が本当にするか読む | `timeout` シナリオに `time.Sleep` なし=即時504(#85) |
| ドキュメント・テンプレート | 実物と diff | `mockport_docs/templates/` が CI 禁止の actions バージョンを案内(#76) |
| 看板(タグライン) | 看板が要求する資産の存在確認 | 「AI-native」に対し llms.txt / AGENTS.md / エージェント導線が0件(#89) |

### レンズB: 同種の並置(重複の発見)

「同じ役割のユニット」を横に並べて比較する。重複は1ファイルの中からは見えない。

- [ ] 同型ユニットを列挙する(今回: 5つのアダプタ、6つの check-*.sh、5つの *-smoke.test.js)
- [ ] 各ユニットの同名・同役割関数を並べる: `grep -n "^func " adapters/*/adapter.go` を見比べる
- [ ] 見つけた重複は **全出現箇所を列挙**してから issue 化(2箇所だけ直して3箇所目を残すのが最悪)
- [ ] **微妙な差分こそ精読する** — 完全一致はただの移動で済むが、微差(line版だけ `defer Body.Close()` がある、githuboauth版だけ Bearer prefix を強制しない)は「どちらを正とするか」の判断が必要で、そこがシニアの仕事

### レンズC: 「能力」vs「使用」(言語・基盤の未活用の発見)

ランタイムのバージョンが提供する機能のチェックリストを作り、コードベースを走査する。

Go の例(`go.mod` のバージョン以下で使えるもの):

| 機能(導入版) | 走査コマンド | 今回の結果 |
|---|---|---|
| `errors.Is/As`(1.13) | `grep -rn "== io.EOF\|err.Error()" ` | 文字列マッチ2箇所、`==`比較4箇所 |
| `any`(1.18) | `grep -rn "interface{}"` | 混在多数 |
| `slices`/`maps`(1.21) | `grep -rn "sort.Strings\|append(\[\]"` | 8+20箇所 |
| `cmp.Or`(1.22) | `if x == "" { x = default }` パターンを目視 | 10箇所以上 |
| ServeMux パターン(1.22) | `grep -rn "case req.Method"` | 5アダプタ全部が手書きルーティング |

このレンズの肝は「**チェックリストを先に作る**」こと。コードを眺めて思い出すのではなく、リストを上から当てる。

### レンズD(プロダクト価値編): ユーザーの動線を歩く

- [ ] **最初の5分を再現する**: README を初見の目で読む → インストール手順はあるか → example をコピペして動くか(今回: 全 example が curl のみで SDK 実アプリ0件 → #86)
- [ ] **比較検索を再現する**: ユーザーが Google する「{competitor} vs」の competitor が比較ページに載っているか(stripe-mock 不在 → #88)
- [ ] **コアループの摩擦を数える**: 主要ユースケース(エラー系テスト)を1周するのに必要な手数(config編集+再起動=摩擦 → #80)
- [ ] **ロードマップの pending を具体化する**: 既存計画の未完了項目は「やると決まっているのに着手単位が無い」状態。issue 化の最有力候補(Phase 25/26/27-28 → #90〜#92)

### 調査の実行形態

- 対象が大きい場合は**並列の読み取り専用エージェント/担当者に領域を分割**して任せる(今回: アダプタ群 / server+cli+config / state+compat+report+security / 非Goアセット の4分割)
- 指示書には「探すもののリスト(上記レンズ)+ 出力形式(ID・severity・file:line・抜粋・修正案)」を明記する。「問題を探して」では人によって出てくるものが変わる

## 4. Stage 2: 証拠固め(spot-check)

調査結果(特に他人・エージェントの報告)を**そのまま issue に書かない**。

- [ ] issue の中核になる指摘は、**自分の目で該当ファイルを開いて行番号と内容を確認**する(今回: stripe routes.go の dead code、ServeMux 移行対象の dispatcher 全部、initLocked、ContractEvidence の3定義とタグを直接確認)
- [ ] 「使われていない」系の指摘は **反証可能な grep** に変換して実行する: `grep -rn "X" --include="*.go" . | grep -v _test` の結果ゼロ、を issue 本文にも確認手順として書く
- [ ] 「同一実装」系の指摘は実物を並べて差分を見る(報告では「完全一致」でも、実際は1行違うことがある — decodePayload の defer がそうだった)
- [ ] 削除・挙動変更を提案する前に、**ゼロ値利用・隠れた呼び出し元・テストの依存**を確認(`Store{}` 複合リテラルの grep をしてから initLocked 削除を書いた)
- [ ] 型の統合を提案する前に、**タグ(json/yaml)と公開スキーマへの影響**を確認(ContractEvidence はタグ完全一致を確認してから alias 案を書いた)
- [ ] 判断に影響する周辺事実も確認(import 循環の有無、CI に staticcheck があるか、など)

**確認していないことは確認していないと書く。** どうしても未確認のまま issue に含める場合は「着手時に◯◯を確認し、結果が△△なら□□に切り替える」と分岐ごと書く(例: @line/bot-sdk の baseURL 対応可否 → 不可なら raw fetch にフォールバック、と #90 に明記)。

## 5. Stage 3: 計画設計

### フェーズ分けはリスク勾配で

```
Phase 1: 挙動不変・機械的(デッドコード削除、stdlib置換)     ← 失敗してもすぐ戻せる。新人の練習に最適
Phase 2: 新規追加のみ(共有ヘルパー新設+置換)               ← 既存挙動に触れない
Phase 3: 構造変更(ルーティング移行、ファイル分割)           ← 挙動変化が出うる。見本issueを最初に1本
Phase 4: 設計判断を伴う整理(型統合、エラー処理)
Phase 5: 横断資産(テスト基盤、CI、docs)
```

- [ ] **見本パターン方式**: 同型作業がN個あるとき、最初の1本(stripe の ServeMux 移行 #66)を「見本」として単独 issue にし、レビューを通してから残りを展開する(#67, #69 は「#66 の実装を読んでから着手」と明記)
- [ ] **依存関係を issue 間で明示**し、並行可能な束を示す(メタ issue に mermaid 図)
- [ ] **粒度は 1 issue = 1 PR = 0.5〜2日**。超えるなら分割(line アダプタは「分割(純移動)」と「移行(ロジック変更)」の2本に割った — 移動と変更を同じ diff に混ぜるとレビュー不能になるため)

### 判断の埋め込みと「見送り表」

- [ ] 分岐がある箇所は**結論+根拠**を書く(「ヘッダ方式を採用。管理APIは並列テストで干渉するため不採用 — この設計判断は変更しないこと」)
- [ ] **検討して見送った選択肢は理由付きでメタ issue に表として残す**。これが無いと、半年後に誰かが同じ検討をやり直すか、見送った理由を知らずに実装してしまう
- [ ] 自分で判断できない事項(公開スキーマの仕様判断など)は issue にせず「メンテナ判断待ち」として明示的に分離する

## 6. Stage 4: 執筆(issueテンプレート)

```markdown
# 概要
何が問題か(1-2段落)+ このissueで何をするか。
> 行番号は YYYY-MM-DD 時点の main(コミットhash)基準。ずれたら各項目の grep で再特定。

## 背景 / なぜ価値があるか
ジュニアが「なぜこれをやるのか」を人に説明できるレベルの解説。
使う言語機能・パターンの最小限の教育的説明(表が有効)。

## 現状の証拠
file:line の表 or リスト + 1-3行の抜粋。再現コマンド(grep等)。

## 設計(この方針に従うこと)
分岐の結論と根拠。「〜は意図的に不採用(理由)」まで書く。

## 作業手順
Step 1, 2, 3...。Before/After コード。コミット分割の指示(「1アダプタ=1コミット」)。
テスト先行(RED→GREEN)が必要な箇所はその旨を明記。
注意点・罠(「◯◯のテストが落ちたら期待値でなく実装を疑え」)。

## 受け入れ基準
- [ ] 機械検証可能な形で(grep ヒット0件 / コマンド成功 / diff ゼロ / 「PR説明に◯◯を貼る」)

## 検証方法
CIと同一のコマンド列 + このissue固有の追加検証。

## スコープ外
やらないことを明示(「ついで修正」の予防)。見送り理由があれば1行。

## 見積もり
0.5〜2日

## 依存関係
先行issue(#番号)と、衝突しうるissueとのマージ順。
```

### 執筆時のセルフチェック(「ジュニアテスト」)

各 issue を書き終えたら自問する:

- [ ] **コンテキストゼロの人が、このissueだけで着手できるか?**(会話の文脈・自分の頭の中にしかない情報を参照していないか)
- [ ] 手順の各 Step に、**迷ったときの判定方法**が書いてあるか(「既存の◯◯を読んで同じ方法を使う」「grep で確認してから」)
- [ ] 挙動が変わる箇所は、**変わってよい範囲**と**変わったら実装ミスの範囲**が区別されているか(404→405は許容・2xxのJSONが変わったらミス、のように)
- [ ] 受け入れ基準に主観語(「きれいに」「適切に」)が残っていないか
- [ ] 「テストの期待値を変更したら PR 説明に全件列挙」のような、**ズルの防止線**があるか
- [ ] 行番号ドリフト対策(基準コミットの明記+再特定コマンド)があるか

## 7. Stage 5: 投稿・配線

- [ ] issue 間の相互参照は、本文をローカルファイルで書く段階では**一意なプレースホルダ**(`ISSUE-1` 等)にし、投稿で番号が確定してから一括置換して `gh issue edit --body-file` で更新する(`#1` のような仮番号は既存 issue/PR と衝突して事故る)
- [ ] **メタ issue(トラッキング)を最後に投稿**: フェーズ別チェックリスト(`- [ ] #N`)、依存図、運用ルール、見送り表、完了の定義
- [ ] 投稿後に全 issue を機械検証: プレースホルダ残存ゼロ、ラベル付与、本文先頭が正しいファイルか(`gh issue view N --json body`)
- [ ] スクリプトで一括投稿する場合、シェルの変数展開の罠に注意(zsh はスカラー変数の無引用展開を分割しない。配列を使うか1コマンドずつ実行する)

## 8. チェックシート(印刷用まとめ)

### A. 調査完了チェック

- [ ] README / ROADMAP / 既存issue / ラベル / ポリシー文書を読んだ
- [ ] 「プロダクトの主張」リストを作り、各主張をコードで突合した(レンズA)
- [ ] 同型ユニットを列挙し、横並びで重複を洗った(レンズB)
- [ ] ランタイム機能チェックリストを作り、未活用を走査した(レンズC)
- [ ] 初見ユーザーの動線(README→install→example)を歩いた(レンズD)
- [ ] CI 定義から検証コマンドの正を採取した
- [ ] 大きな対象は領域分割し、調査指示に「探すもの+出力形式」を明記した

### B. 証拠チェック(issue 1本ごと)

- [ ] 中核の指摘を自分の目で file:line 確認した
- [ ] 「未使用」は反証可能な grep で確認し、コマンドを本文に載せた
- [ ] 削除・統合の前提(ゼロ値利用、タグ、import循環、テスト依存)を確認した
- [ ] 未確認事項は「着手時に確認+分岐」として明記した

### C. 品質チェック(issue 1本ごと)

- [ ] 1 PR / 0.5〜2日に収まる粒度
- [ ] 設計分岐は結論+根拠で決め切ってある(不採用案も)
- [ ] Before/After コードがある
- [ ] 受け入れ基準がすべて機械検証可能
- [ ] 検証コマンドが CI と同一+固有検証
- [ ] スコープ外・依存関係・基準コミット・再特定コマンドがある
- [ ] ジュニアテスト(コンテキストゼロで着手可能か)を通した

### D. 計画全体チェック

- [ ] フェーズがリスク勾配順(挙動不変→追加→構造変更→判断→横断)
- [ ] 同型作業は見本1本を先行させる構成
- [ ] 依存図と並行可能な束が示されている
- [ ] 見送り表(理由付き)がメタ issue にある
- [ ] メタ issue に運用ルール(1 issue=1 PR、ついで修正禁止、期待値変更の申告義務)と完了の定義がある

## 9. アンチパターン(やってしまいがちな失敗)

| アンチパターン | 何が起きるか | 対策 |
|---|---|---|
| 「◯◯を改善する」だけの issue | 実装者ごとに別物が出てくる | Before/After と受け入れ基準で結果を固定 |
| 調査報告の転記(裏取りなし) | 行番号違い・存在しない関数で実装者が信頼を失う | Stage 2 の spot-check を必須化 |
| 重複の部分修正 | 2/5 箇所だけ直り、残りが「直した版」と乖離 | 全出現箇所の列挙を issue に含める |
| 移動と変更を同じ issue に | diff がレビュー不能になり、バグ混入に気づけない | 「純移動」と「変更」を別 issue/コミットに分離 |
| 善意の「ついで修正」許容 | PR が膨らみ、無関係なリグレッションが混ざる | 運用ルールで禁止し「気づいたら新 issue」へ誘導 |
| 主観的な完了条件 | 「できた」の水準が人によって違う | grep/コマンド/diff で機械判定できる形に書き直す |
| 新しい検証手段の発明 | CI と乖離し、ローカルでは通るのに CI で落ちる | 検証は CI のコマンド列を正とする |
